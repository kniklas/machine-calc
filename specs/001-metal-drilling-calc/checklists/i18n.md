# Specification Quality Checklist: Internationalization (i18n) Requirements

**Purpose**: Validate the completeness, clarity, consistency, and testability of the internationalization/localization requirements (message catalog, locale selection, English-only logging) before `/speckit.implement` executes the i18n tasks (T009a, T009b, T018a, T026a, T026b, T033a, T043a).
**Created**: 2026-07-10
**Feature**: [spec.md](../spec.md)
**Depth**: Standard (pre-implementation self-review)
**Audience**: Author (self-review before `/speckit.implement`)

## Requirement Completeness

- [ ] CHK001 - Are requirements defined for every user-facing surface that must be localized (REPL prompts, labels, validation/error messages, feasibility warnings)? [Completeness, Spec §FR-019]
- [x] CHK002 - Is there an explicit requirement for what the library API returns when no `locale` is supplied at all (vs. an empty string or unsupported value)? [Gap, Spec §FR-019] — **Resolved**: FR-019d now states an empty string is treated the same as omitting `locale` (defaults to English); contracts/library-api.md updated to match.
- [ ] CHK003 - Are requirements defined for how a new locale is added by a contributor (file naming, discovery/registration mechanism), or only for the runtime fallback behavior? [Gap, Spec §FR-019, SC-007]
- [ ] CHK004 - Are requirements defined for what happens if `MACHINE_CALC_LOCALE` is set to a syntactically valid but completely unsupported locale code (e.g. `"xx-YY"`)? [Edge Case, Spec §FR-019]
- [ ] CHK005 - Is it specified whether catalog completeness (i.e., every message key present in every shipped locale) is validated at all, or only relied upon via runtime fallback? [Gap]

## Requirement Clarity

- [ ] CHK006 - Is "stable, language-independent identifier" (message key) defined precisely enough to be objectively checked (e.g., naming convention, uniqueness rule)? [Clarity, Spec §FR-019]
- [x] CHK007 - Is "unrecognized" (as in "falling back to English if unset or unrecognized") defined — does it mean "no matching catalog module exists," or something broader (e.g., malformed value)? [Ambiguity, Spec §FR-019] — **Resolved**: FR-019c now defines "unset or unrecognized" as "absent, empty, or does not match the name of any locale module bundled with the installation."
- [ ] CHK008 - Is the boundary between "message catalog" (FR-019) and "application/diagnostic logging" (also FR-019, English-only) drawn clearly enough that a specific log statement's classification is unambiguous? [Clarity, Spec §FR-019]
- [x] CHK009 - Is the placeholder substitution rule (`str.format()`-style named placeholders) specified precisely enough to know what happens if a required placeholder value is missing at lookup time? [Gap, Spec §FR-019] — **Resolved**: FR-019b now requires the lookup to never raise to the caller, with any formatting failure logged (English) rather than surfaced as an exception; T009a/T018a updated to test this.

## Requirement Consistency

- [ ] CHK010 - Are the locale-selection requirements consistent between the interactive text interface (env-var-only, FR-002/FR-019) and the library API (explicit `locale` parameter, FR-019) — i.e., is it clear these are two independent, non-conflicting selection mechanisms for the same underlying catalog? [Consistency, Spec §FR-002, §FR-019]
- [ ] CHK011 - Do FR-009, FR-010, FR-013, and FR-015 (which each separately reference "sourced from the message catalog per FR-019") use consistent language/terminology for the same underlying mechanism? [Consistency, Spec §FR-009, §FR-010, §FR-013, §FR-015]
- [ ] CHK012 - Is the English-fallback behavior described identically across FR-019, the Message Catalog entity (data-model.md), and the Localization contract (contracts/library-api.md), with no contradictory wording about when fallback occurs? [Consistency]

## Acceptance Criteria Quality

- [x] CHK013 - Is SC-007 ("a new supported language can be added ... with zero changes to calculation or validation logic") measurable/verifiable as written, or does it depend on an undefined notion of "calculation or validation logic" boundaries? [Measurability, Spec §SC-007] — **Resolved**: SC-007 now names the specific modules (`operations/*/formulas.py`, `validation.py`, `operations/*/__init__.py`) that must be unchanged.
- [ ] CHK014 - Is there a measurable acceptance criterion for the "identical text for the same catalog" guarantee between CLI and library output (referenced in tasks.md T026a/T033a) that traces back to a spec-level requirement or success criterion? [Traceability, Gap]
- [ ] CHK015 - Are there acceptance criteria for locale selection specifically (e.g., "CLI output matches catalog X when `MACHINE_CALC_LOCALE=X`"), distinct from the general message-catalog acceptance criteria? [Gap, Spec §FR-019]

## Scenario Coverage

- [ ] CHK016 - Does quickstart.md's Scenario 8 (non-default locale) cover the *library* path only, or is there an equivalent documented scenario for the *CLI* path (env-var-driven) as well? [Coverage, Spec §Quickstart Scenario 8]
- [ ] CHK017 - Is there a documented scenario for a *partially* translated non-English catalog (some keys present, some missing) exercising the per-key fallback distinctly from the per-locale (whole-catalog-missing) fallback? [Coverage, Edge Case]
- [x] CHK018 - Is there a documented scenario confirming that changing `MACHINE_CALC_LOCALE` mid-REPL-session (FR-014's "loop to allow further calculations") does or does not change the active locale within that same running session? [Coverage, Gap, Spec §FR-014] — **Resolved**: FR-019c now states the locale is fixed for the lifetime of the session and not re-read mid-loop; data-model.md and T026a updated to match.

## Non-Functional Requirements

- [x] CHK019 - Is there a requirement or constraint capping the runtime/memory cost of loading message catalogs, consistent with Constitution Principle V's resource-constrained targets? [Gap, Spec §Assumptions] — **Resolved**: new Assumptions bullet states catalogs are expected to remain small, dependency-free Python modules consistent with Principle V.
- [x] CHK020 - Is it specified whether adding additional locale catalogs in the future is expected to remain dependency-free (per the "no gettext/JSON parser" decision in research.md #9), or could a future locale require a new dependency? [Ambiguity, Assumption] — **Resolved**: same new Assumptions bullet states this expectation is intended to hold for future locales too.

## Dependencies & Assumptions

- [x] CHK021 - Is the assumption that only the English catalog ships with this feature (spec.md Assumptions) clearly flagged as a scope boundary rather than an oversight, so reviewers don't mistake it for incomplete work? [Clarity, Spec §Assumptions] — **Resolved**: Assumptions bullet reworded to explicitly say "an intentional scope boundary, not a partial or incomplete implementation of FR-019."
- [ ] CHK022 - Is the dependency of the library's localized `ErrorInfo.message` on the same catalog module used by the CLI made explicit enough to prevent the two from silently diverging over time (e.g., duplicate/forked catalogs)? [Consistency, Dependency]

## Ambiguities & Conflicts

- [x] CHK023 - Is there any conflict between FR-019's "MUST NOT" language around OS-locale auto-detection and any pre-existing assumption elsewhere in spec.md about locale/environment behavior? [Conflict check, Spec §FR-019] — **Verified no conflict**: no other spec.md section references OS/system locale.
- [x] CHK024 - Is a requirement/acceptance-criteria ID scheme established for locale-specific behaviors (e.g., tying CHK-style traceability back to FR-019 sub-clauses individually), given FR-019 bundles multiple distinct rules (catalog format, env var, library param, fallback, logging exclusion) into a single requirement ID? [Traceability, Gap] — **Resolved**: FR-019 split into FR-019a (catalog format), FR-019b (placeholders), FR-019c (interactive locale selection), FR-019d (library locale param), FR-019e (fallback), FR-019f (logging exclusion); tasks.md T009a updated to cite specific sub-IDs.
